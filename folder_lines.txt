EQC Asset Management Platform
Frontend — Complete Folder &amp; File Reference
OTH Equipment Co. | Olympus Thailand Medical Division  •  Version 1.1  •  May 2026
Overview
This document lists every file and folder in the EQC frontend React application, derived directly from EQC_Frontend_Technical_Spec.md (v1.1, May 2026). Each entry explains what the file does, what it owns, and why it exists. Use this as the authoritative reference when generating, navigating, or reviewing the codebase.
The application is a React 18 / TypeScript SPA built with Vite, authenticated via Azure AD MSAL, and communicates exclusively with the EQC backend REST API. The folder structure follows feature-first grouping within src/ so that every page module is self-contained and shared code is clearly separated into components/, hooks/, api/, types/, and utils/.
Complete Folder Tree
The full structure is presented below, section by section. Files are colour-coded in the tables: the File column uses monospace green text for file names.
Root Level — Configuration Files
These files live at the project root and control the build, tooling, and deployment pipeline. They are not inside src/.
File
Purpose
.env.example
Template listing every required environment variable. Committed to source control. Developers copy this to .env.local and fill in values. Real credentials are never committed.
.gitignore
Git ignore rules. Excludes: node_modules/, dist/, .env.local, coverage/, *.log files.
index.html
Vite entry HTML. Contains the &lt;div id=&apos;root&apos;&gt; mount point and the &lt;script type=&apos;module&apos; src=&apos;/src/main.tsx&apos;&gt; entry reference. No inline scripts.
vite.config.ts
Vite configuration. Sets @ path alias to ./src. Enables React plugin. Configures proxy for /api to the backend in development (avoids CORS issues). Sets build output to dist/.
tsconfig.json
TypeScript strict mode config. Path aliases: @ → src/. target: ES2020. moduleResolution: bundler. jsx: react-jsx.
tsconfig.node.json
TypeScript config for the Vite config file itself (Node.js context, separate from the browser build).
tailwind.config.ts
Tailwind configuration. Extends default theme with Olympus brand colours (primary blue, accent teal, neutral grays). Configures content paths for purging unused classes.
package.json
Project manifest. Scripts: dev (vite), build (tsc &amp;&amp; vite build), preview (vite preview), test (vitest), lint (eslint), format (prettier). Lists all dependencies and devDependencies.
eslint.config.ts
ESLint flat config. Uses @typescript-eslint/recommended, plugin:react-hooks/recommended, plugin:jsx-a11y/recommended. Enforces no console.log in production files.
staticwebapp.config.json
Azure Static Web Apps routing rules. Routes all paths to index.html for SPA client-side routing. Sets security headers (CSP, HSTS, X-Frame-Options). Configures MIME types.

Purpose
main.tsx
Application entry point. Creates the MSAL PublicClientApplication from msalConfig. Wraps the React tree in MsalProvider, BrowserRouter, and QueryClientProvider. Renders &lt;App /&gt;.

Type Definitions — src/types/
All TypeScript interfaces and union types for the application. These are derived directly from the EQC_Objects_and_Attributes.xlsx schema (v2.0). When the backend schema changes, update these files first — TypeScript will surface every usage that needs updating.
File
Purpose
auth.types.ts
TypeScript interface for the authenticated user object: AuthUser { id, email, name, role: UserRole, sfdc_user_id? }. The role field uses the UserRole union type defined in enums.ts.
enums.ts
All application string-literal union types mirroring the backend schema (not TypeScript enums — raw string unions for direct API comparison). Includes: UserRole (8 values), AssetStatus (15 values), AssetDemoLoanerType (9 values), AssetConditionGrade, AssetAgeGroup, FDAStatus, AnnualInspectionStatus, SalesRequestStatus (11 values), SalesRequestRecordType, SalesRequestPurpose1/2, ExtensionStatus, DeploymentStatus/Type, BillingCycle, DispatchDocumentType/Status, InspectionResult/Type, OverallCondition, RepairCaseStatus/Type, AreaCode, EventEntityType, EventType, ActorType, AlertDeliveryStatus, AIPredictionType.
api.types.ts
Response shape interfaces: ApiResponse&lt;T&gt;, PaginatedResponse&lt;T&gt;, ApiError { code, message, details }. Used by all query hooks to correctly type API responses.
asset.types.ts
Interface for the assets table: Asset (all 30+ fields including asset_id, status, demo_loaner_type, condition_grade, fda_status, etc.), CreateAssetPayload, UpdateAssetPayload.
salesRequest.types.ts
Interface for the sales_requests table — the primary operational object. SalesRequest (all fields including request_number like DR-2602-106504, record_type, status, purpose1/2, dates, extension_count, bom_set_id, etc.), CreateSalesRequestPayload.
extension.types.ts
Interfaces for the request_extensions table: RequestExtension (extension_id, parent_request_id, new_return_date, reason_code, status, etc.), CreateExtensionPayload.
bom.types.ts
Interfaces for bom_sets, bom_line_items, and accessory_master tables: BOMSet, BOMLineItem (with is_required dispatch-block flag), AccessoryMaster (accessory_code like MAJ-1435), PackingValidationResult { isComplete, missingItems[] }.
deployment.types.ts
Interface for the device_deployments table (the asset-to-request link record): DeviceDeployment (deployment_id, request_id, asset_id, deployment_type, status, start_date, expected_return_date, days_outstanding computed field, condition_on_dispatch/return, billing fields, etc.).
dispatch.types.ts
Interface for the dispatch_documents table: DispatchDocument (doc_id, deployment_id, document_type, pdf_blob_url SAS URL, qr_code_value/image_url, signer details, status, sap_gi_triggered flags, etc.).
inspection.types.ts
Interfaces for inspection_records and inspection_line_items tables: InspectionRecord (inspection_id, deployment_id, overall_condition, inspected_by_id, etc.), InspectionLineItem (item_id, bom_line_id, result PASS/FAIL/MISSING, quantity_actual, inspection_type DISPATCH/RETURN).
repair.types.ts
Interface for the repair_cases table: RepairCase (repair_id, rs_number like RS-202602-099595, eas_no, asset_id, account_id, sfdc_repair_id, status, repair_type, area, repair_cost_thb, fse_assigned_id, created_at).
account.types.ts
Interface for the accounts table: Account (account_id, account_name, address, area, department, segmentation, group_wave, created_at).
eventLog.types.ts
Interface for the event_log table (immutable audit trail): EventLog (log_id, entity_type, entity_id, event_type, old_value, new_value JSON strings, actor_id, actor_type, timestamp, narrative pre-formatted string, ip_address, session_id).
user.types.ts
Interface for the users table: User (user_id, name, email, role: UserRole, team, area, sfdc_user_id, is_active).
report.types.ts
ReportType union, ReportFilter, ReportRow, ScheduledReport interfaces. Covers 7 report types: SalesRequestSummary, LoanerBilling, InspectionDefects, AssetUtilization, OverdueAnalysis, BOMCompliance, RepairCaseSummary.
dashboard.types.ts
Response shape interfaces for each dashboard API endpoint: SalesDashboardData, EQCDashboardData, InventoryDashboardData, OverdueFeedData, ExecutiveDashboardData, FinanceDashboardData.
API Layer — src/api/
All backend communication is centralised here. Each file targets one database entity or domain. Components and hooks never call Axios directly — they always go through these functions. This means API changes only require updating one file per entity.
File
Purpose
axiosInstance.ts
Axios instance with base URL from VITE_API_BASE_URL. Request interceptor: acquires MSAL token silently, attaches Authorization: Bearer {token} header. Response interceptor: normalises errors into ApiError shape. On 401, clears auth state and redirects to /login.
auth.api.ts
getMe(): Promise&lt;AuthUser&gt; — calls GET /api/auth/me. Used on app load to fetch the authenticated user&apos;s profile and role from the backend.
assets.api.ts
API functions for the assets table. listAssets(filters), getAsset(id), createAsset(payload), updateAsset(id, payload), transitionAssetStatus(id, status). Filters: status, model_code, warehouse_code, demo_loaner_type, business_unit, area, is_active.
salesRequests.api.ts
API functions for the sales_requests table — the primary request object. listSalesRequests(filters), getSalesRequest(id), createSalesRequest(payload), approveSalesRequest(id), rejectSalesRequest(id, reason), markReturned(id), cancelSalesRequest(id). Filters: status, record_type, purpose1/2, account_id, sales_person_id, date_range.
extensions.api.ts
API functions for the request_extensions table. createExtension(payload), listExtensions(parent_request_id), approveExtension(id), rejectExtension(id).
deployments.api.ts
API functions for the device_deployments table (the asset-to-request link). listDeployments(filters), getDeployment(id), createDeployment(request_id, asset_id, payload), transitionDeploymentStatus(id, status). Note: most operational actions are taken at the SalesRequest level; the deployment is updated as a backend side-effect.
bom.api.ts
API functions for bom_sets, bom_line_items, and accessory_master. listBOMSets(model_code?), getBOMSet(id), createBOMSet(payload), updateBOMSet(id, payload), getBOMLineItems(set_id), createBOMLineItem(set_id, payload), updateBOMLineItem(line_id, payload), listAccessoryMaster(filters), validatePacking(bom_set_id, packed_line_ids[]) → PackingValidationResult.
dispatch.api.ts
API functions for dispatch_documents. generateDispatchDocument(request_id, document_type), getDispatchDocument(doc_id), listDispatchDocuments(filters), markSentToPrint(doc_id), uploadSignedCopy(doc_id, file: File, signed_by_name). Filters: status, deployment_id, date_range.
inspection.api.ts
API functions for inspection_records and inspection_line_items. createInspection(deployment_id), getInspection(id), recordLineItemResult(inspection_id, line_id, result, quantity_actual?, notes?), completeInspection(inspection_id, overall_condition, notes?).
repairCases.api.ts
API functions for repair_cases. listRepairCases(filters), getRepairCase(repair_id), updateRepairCase(repair_id, payload). Filters: status, asset_id, account_id, area, repair_type. Repair cases are auto-created by the backend on inspection FAIL — no create function on the frontend.
dashboard.api.ts
getSalesDashboard(), getEQCDashboard(), getInventoryDashboard(), getOverdueFeed(), getExecutiveDashboard(), getFinanceDashboard(). Each returns a strongly-typed dashboard data object matching the types in dashboard.types.ts.
reports.api.ts
getReportData(type: ReportType, filters), exportReport(type, filters, format: xlsx|csv|pdf), scheduleReport(payload), listScheduledReports(), deleteScheduledReport(id).
accounts.api.ts
API functions for the accounts table. listAccounts(filters), getAccount(id), createAccount(payload), updateAccount(id, payload). Filters: area, segmentation, group_wave.
users.api.ts
API functions for the users table (ADMIN only). listUsers(), getUser(id), createUser(payload), updateUserRole(id, role: UserRole), deactivateUser(id). Integration_Service role is system-only and cannot be assigned via UI.
eventLog.api.ts
API functions for the event_log table. getEventLog(entity_type, entity_id, pagination) — paginated EventLog[] for the entity timeline. getEventLogByUser(user_id, pagination) — for admin audit views. The event log is immutable — no create/update/delete from the frontend.
serviceContracts.api.ts
listServiceContracts(asset_id?), getServiceContract(contract_id). Read-only from the frontend — contracts are managed by admins or integrated via SAP.
Custom Hooks — src/hooks/
TanStack Query (React Query) hooks that wrap the api/ functions with server-state management, caching, background refresh, and optimistic update patterns. Components consume these hooks rather than calling api/ functions directly.
File
Purpose
useAuth.ts
Wraps useMsal() and the AuthUser context. Returns { user, isLoading, isAuthenticated, login, logout }. login() calls MSAL loginPopup(). logout() calls logoutPopup(). After successful login, fetches backend user profile via getMe().
useCurrentUser.ts
Returns the AuthUser from AuthContext. Throws if called outside AuthProvider. Used throughout the app to access user.role for RBAC rendering decisions.
useHasRole.ts
useHasRole(...roles: UserRole[]): boolean — returns true if the current user&apos;s role is in the allowed list. Used for conditional rendering of buttons and UI sections.
useAssets.ts
TanStack Query hooks for the assets table: useAssets(filters), useAsset(id), useCreateAsset(), useUpdateAsset(), useTransitionAssetStatus(). All mutations invalidate [&apos;assets&apos;] and relevant dashboard caches on success.
useSalesRequests.ts
TanStack Query hooks for the sales_requests table: useSalesRequests(filters), useSalesRequest(id), useCreateSalesRequest(), useApproveSalesRequest(), useRejectSalesRequest(), useMarkReturned(), useCancelSalesRequest(). All mutations invalidate [&apos;salesRequests&apos;] and relevant dashboard caches.
useExtensions.ts
useExtensions(parent_request_id), useCreateExtension(), useApproveExtension(), useRejectExtension(). Mutations invalidate the parent SalesRequest query.
useDeployments.ts
TanStack Query hooks for the device_deployments table: useDeployments(filters), useDeployment(id). Primarily used for reading deployment details and history — most state changes flow from useSalesRequests mutations.
useBOM.ts
useBOMSets(model_code?), useBOMSet(set_id), useBOMLineItems(set_id), useAccessoryMaster(filters), useCreateBOMSet(), useUpdateBOMSet(), useCreateBOMLineItem(), useUpdateBOMLineItem(), useValidatePacking().
useDispatch.ts
useDispatchDocuments(filters), useDispatchDocument(doc_id), useGenerateDispatchDocument(), useMarkSentToPrint(), useUploadSignedCopy().
useInspection.ts
useInspection(inspection_id), useCreateInspection(), useRecordLineItemResult(), useCompleteInspection().
useRepairCases.ts
useRepairCases(filters), useRepairCase(repair_id), useUpdateRepairCase().
useDashboard.ts
useSalesDashboard(), useEQCDashboard(), useInventoryDashboard(), useOverdueFeed(), useExecutiveDashboard(), useFinanceDashboard(). All use staleTime: 5 min and refetchInterval: 5 min for auto-refresh per solution design spec.
useReports.ts
useReportData(type, filters), useExportReport(), useScheduledReports(), useCreateScheduledReport(), useDeleteScheduledReport().
usePagination.ts
Generic hook managing page, limit, setPage, setLimit state. Used alongside TanStack Table for server-side pagination across all list pages.
useDebounce.ts
useDebounce&lt;T&gt;(value: T, delay: number): T — debounces search input before firing API calls. Used in SearchInput components to avoid excessive API requests on every keystroke.
Context Providers — src/context/
File
Purpose
AuthContext.tsx
React context holding the authenticated AuthUser object. AuthProvider wraps the app inside MsalProvider. On mount, calls getMe() if MSAL reports an active account. Exposes user, setUser, isLoading. Consumed via the useCurrentUser() hook.
ThemeContext.tsx
Light/dark mode context (defaults to light). Reads preference from localStorage. Applies the dark class to the &lt;html&gt; element for Tailwind dark mode variants.
Layouts — src/layouts/
Shell components that provide the persistent UI frame (sidebar, nav bar) for authenticated pages and a minimal frame for public pages.
File
Purpose
AuthenticatedLayout.tsx
Shell layout for all authenticated pages. Renders &lt;Sidebar /&gt;, &lt;TopNav /&gt;, and the &lt;main&gt; content area with &lt;Outlet /&gt; for nested routes. Fetches current user on mount. Redirects to /login if not authenticated.
PublicLayout.tsx
Minimal layout for unauthenticated pages (login, error pages). Renders only the Olympus logo and content area — no sidebar or navigation.
Routing — src/router/
All route configuration, protected route logic, and role-based dashboard redirection live here. Pages never need to know about routing beyond their own path — all navigation logic is centralised here.
File
Purpose
index.tsx
Main route configuration using createBrowserRouter. Defines all top-level routes: /login → LoginPage, / → AuthenticatedLayout with nested routes for /dashboard, /sales-requests, /assets, /bom, /dispatch, /inspections, /reports, /accounts, /users, /audit, * → NotFoundPage.
ProtectedRoute.tsx
Wrapper component. Checks useIsAuthenticated(). If false, redirects to /login with the current path in location state. If authenticated but role not in allowedRoles prop, renders UnauthorizedPage.
RoleRedirect.tsx
At /dashboard, resolves which sub-dashboard to redirect to based on user.role: Sales_Rep/FSE → /dashboard/sales, EQC_Operator → /dashboard/eqc, EQC_Manager/Sales_Manager → /dashboard/manager, Executive/System_Admin → /dashboard/executive, Integration_Service → /dashboard/inventory.
Pages — src/pages/
Each page is an isolated module folder. Pages do not import from other page folders — shared code lives in components/, hooks/, or utils/. This ensures that changing one page never breaks another.
